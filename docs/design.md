## Problem Statement

30~40명 규모의 팀이 미국 OTC 선케어 제품 개발 프로세스를 엑셀/구글시트로 관리 중. 복합적인 문제 발생:
- 전체 진행상황을 한눈에 파악하기 어려움
- 파일(시험성적서, CT 결과 등) 관리가 분산됨
- 제품이 늘어나면 시트가 복잡해져서 안 봄
- 누가 언제 뭘 바꿨는지 추적 불가 (감사 이력 부재)
- 경영진이 전체 현황을 보고 싶어함

## Demand Evidence

- 팀이 이미 엑셀로 프로세스를 관리하고 있음 → 문제 인식과 워크플로우가 존재
- 프로토타입을 직접 만들 정도의 내부 동기가 있음
- 30~40명이 사용할 도구 → 조직적 수요

## Status Quo

하나의 마스터 엑셀 파일을 공유해서 사용 중. 컬럼 구조는 일정하나, 엑셀의 근본적 한계:
- 동시 편집 시 충돌
- 변경 이력 추적 불가
- 파일 첨부 불가
- 대시보드/요약 뷰 없음
- 권한 구분 없음

## Target User & Narrowest Wedge

**Primary:** 제형담당자, 영업담당자 (매일 단계 상태 업데이트)
**Secondary:** 경영진/팀장 (전체 현황 조회, 읽기 전용)

Wedge: "엑셀 대신 이걸 열면 내 담당 제품의 현재 상태와 다음 할 일이 바로 보인다"

## Constraints

- 사내 서버 1대에 배포 (클라우드 없음)
- 30~40명 동시 접속
- 기존 엑셀 데이터 마이그레이션 필요
- 로그인/권한 필수 (viewer/editor)

## Premises

1. **DB는 PostgreSQL** — 30~40명 동시 접속, SQLite로는 write lock 문제 발생
2. **로그인 + 권한 관리 필요** — viewer(읽기 전용) / editor(수정 가능) 역할 구분
3. **엑셀 임포트 포맷은 고정** — 하나의 마스터 엑셀을 공유해서 쓰고 있어 컬럼 구조 일정
4. **변경 이력은 단계 상태 변경만 추적** — 모든 필드 변경까지는 불필요
5. **사내 서버 1대면 충분** — 온프레미스 배포

## Cross-Model Perspective

Claude subagent의 독립적 분석:

**Steelman:** 이건 규제 산업의 컴플라이언스 핵심 워크플로우 도구다. OTC 선케어 제품은 FDA 문서, 임상시험 기록, 제조 감사 추적이 필요하다. 감사 이력이 없는 엑셀은 규제 감사가 가장 중요한 시점에 정확히 무너진다.

**핵심 인사이트:** "no audit trail"이 진짜 문제. 상태 파악, 파일 관리, 복잡도는 모두 증상이고, 근본 원인은 누가 뭘 승인했는지 증명할 수 없다는 것.

**도전한 전제:** 엑셀 포맷 고정 전제를 도전했으나, 사용자가 마스터 엑셀 공유 방식임을 확인 → 전제 유지.

**프로토타입 우선순위:** Auth → 감사 이력 → 대시보드 순. 감사 추적 + 인증이 돌이킬 수 없는 아키텍처 결정이므로 먼저 맞추고 나머지를 쌓아라.

## Approaches Considered

### Approach A: 현재 코드 확장
Summary: 지금 Next.js + Prisma 구조 위에 5가지 기능 추가. DB만 SQLite→PostgreSQL 마이그레이션.
Effort: M (Claude Code로 1~2시간)
Risk: Low
Pros: 이미 동작하는 코드 기반, 최소 변경, 빠른 출시
Cons: NextAuth 추가로 복잡도 증가, 감사 이력을 기존 스키마에 붙이면 깔끔하지 않음
Reuses: 기존 UI 컴포넌트 전체, API 라우트, Prisma 스키마

### Approach B: 깨끗한 재설계 (선택됨)
Summary: 인증/권한/이력 추적을 처음부터 고려해 DB 스키마와 API를 재설계. 기존 UI 컴포넌트는 재사용.
Effort: L (Claude Code로 2~3시간)
Risk: Med
Pros: 감사 이력이 아키텍처에 내장, 권한 관리 자연스러움, 향후 확장 용이
Cons: 시간 더 걸림, 기존 API 라우트 재작성 필요
Reuses: UI 컴포넌트(ProductList, ProductDetail, StatusBadge, ProgressBar, Navbar), Tailwind 스타일링

## Recommended Approach

**Approach B: 깨끗한 재설계**

30~40명이 쓰는 도구에서 감사 이력과 권한은 나중에 붙이기 어렵다. 아키텍처 기반을 단단히 가져가는 게 맞다.

### 기술 스택

| 영역 | 기술 | 이유 |
|------|------|------|
| Frontend | Next.js 16 + React 19 + Tailwind CSS 4 | 기존 유지 |
| Backend | Next.js API Routes + Prisma 7 | 기존 유지 |
| DB | PostgreSQL | 동시 접속 30~40명 지원 |
| 인증 | Auth.js v5 (Credentials Provider) | 사내 도구, 심플한 ID/PW 인증. Next.js 16 App Router 호환 확인 필요 (node_modules/next/dist/docs/ 참조) |
| 파일저장 | 로컬 파일시스템 (uploads/) | 기존 유지, 사내 서버 |

### DB 스키마 변경

```
기존 유지:
- Product (기본 정보 + 처방개발 + 임상/시험 + 시험생산 + 양산/출고 필드)
- ProcessStep (productId, stepKey, status, note, dueDate, completedDate)
- StepFile (stepId, fileName, originalName, fileSize, uploadedAt)

추가:
- User (id, email, name, password, role: 'viewer'|'editor'|'admin', createdAt)
- StepHistory (id, stepId, productId, oldStatus, newStatus, changedBy, changedAt)
  → immutable table, INSERT only, no UPDATE/DELETE (앱 레벨: Prisma에서 update/delete 호출 금지. DB 레벨: PostgreSQL에서 DELETE/UPDATE 트리거로 RAISE EXCEPTION 추가 권장)
- Session (NextAuth 세션 관리)
```

### 구현 순서 (우선순위)

**Phase 1: MVP Core (런치 필수)**
1. **Next.js 16 호환성 확인** — node_modules/next/dist/docs/ 읽고 Auth.js v5 + App Router 조합 검증
2. **PostgreSQL 마이그레이션 + User 모델** — DB 전환, 사용자 테이블 생성
3. **인증 (Auth.js v5)** — 로그인/로그아웃, 세션 관리, viewer/editor/admin 역할. 비밀번호 리셋은 admin이 CLI 또는 admin UI에서 초기화.
4. **권한 미들웨어** — editor만 수정 가능, viewer는 읽기 전용
5. **StepHistory 테이블 + 감사 이력** — 단계 상태 변경 시 자동 기록, 변경 이력 조회 UI
6. **대시보드/요약 뷰** — 전체 제품 진행률, 단계별 완료/진행/지연 카운트
7. **검색/필터** — 고객사(customer), 개발팀(devTeam), 상태, 제형담당(formulator)별 필터링 (모두 Product 모델에 이미 존재하는 필드)
8. **알림/리마인더** — 목표일 초과 단계 하이라이트 (빨간 배지), 대시보드에 지연 항목 섹션. **앱 내 시각 표시만.** 이메일/Slack 알림은 Out of Scope.

**Phase 2: 런치 후 1주 내**
9. **엑셀 임포트** — 엑셀 파일 샘플 확보 후 구현. xlsx 파싱, 컬럼 매핑, 벌크 INSERT, 에러 리포트. 샘플 미확보 시 수동 입력 또는 prisma seed 스크립트로 초기 데이터 마이그레이션.

### 대시보드 뷰 구성

- **전체 요약:** 총 제품 수, 완료/진행중/지연 카운트
- **제품별 진행률 바:** 20단계 중 몇 단계 완료인지 시각화
- **지연 항목 하이라이트:** 목표일 초과 + 미완료 단계 목록 (빨간 배지)
- **팀별/고객사별 그룹 뷰:** 필터링 가능

### 권한 모델

| 역할 | 제품 조회 | 상태 변경 | 파일 업로드 | 제품 등록/삭제 | 사용자 관리 |
|------|----------|----------|------------|--------------|------------|
| viewer | O | X | X | X | X |
| editor | O | O | O | X | X |
| admin | O | O | O | O | O |

### 엑셀 임포트 설계

- xlsx 파일 업로드 → 서버에서 파싱 (SheetJS/xlsx 라이브러리)
- 컬럼 매핑: 엑셀 헤더 → DB 필드 자동 매칭
- 미리보기: 임포트 전 파싱 결과 테이블로 보여줌
- 에러 처리: 필수 필드 누락, 타입 불일치 시 행 번호와 함께 에러 표시
- 트랜잭션: 전체 성공 또는 전체 롤백

### 파일 업로드 제한

- 최대 파일 크기: 50MB (OTC 규제 PDF 문서 대응)
- 허용 확장자: .pdf, .xlsx, .xls, .doc, .docx, .jpg, .png
- 저장 경로: uploads/{productId}/{stepKey}/

### 사용자 프로비저닝

- 초기 셋업: admin 계정 1개를 seed 스크립트로 생성
- 이후 사용자 추가: admin이 사용자 관리 UI에서 계정 생성
- 비밀번호 리셋: admin이 사용자 관리 UI에서 초기화

3. 대시보드에 차트(파이, 바)가 필요한가? 테이블만으로 충분한가?

## Success Criteria

- [ ] 30~40명이 동시 접속해도 응답 시간 2초 이내
- [ ] viewer는 어떤 수정도 불가능함을 확인
- [ ] 단계 상태 변경 시 StepHistory에 자동 기록됨
- [ ] 경영진이 대시보드에서 전체 진행 현황을 10초 이내에 파악 가능
- [ ] 기존 엑셀 데이터가 임포트되어 기존 데이터 손실 없음
- [ ] 목표일 초과 단계가 시각적으로 즉시 식별됨

## Distribution Plan

- 사내 서버 1대에 직접 배포
- Docker Compose 권장 (PostgreSQL + Next.js App 컨테이너 격리, 메모리 경쟁 방지)
- fallback: `npm run build && npm start` + 별도 PostgreSQL 서비스
- 최소 권장 서버 스펙: RAM 4GB+, 디스크 20GB+ (파일 업로드 고려)
- 사내 네트워크에서 IP:PORT 또는 내부 도메인으로 접근

## Dependencies

- PostgreSQL 서버 설치/접근 권한
- 사내 서버 접근 권한 (배포용)
- 기존 엑셀 파일 샘플 (임포트 개발/테스트용)
- 초기 사용자 계정 목록 (30~40명)

## The Assignment

**팀에게 지금 쓰고 있는 엑셀 파일 원본 1개를 받아라.** 정리하지 말고 있는 그대로. 이 파일이 엑셀 임포트 기능의 정확한 스펙이 된다. 컬럼 헤더, 데이터 형식, 빈 셀 패턴 전부. 파일 없이 임포트를 만들면 나중에 다시 만든다.

## What I noticed about how you think

- 처음에 "MVP를 빨리 완성하는 게 목표"라고 했지만, 접근 방식에서 B(재설계)를 골랐다. 속도보다 기반의 단단함을 선택한 것. 30~40명이 쓸 도구라면 맞는 판단이다.
- 전제 확인에서 팀 규모(30~40명)와 권한 관리를 명확히 짚었다. "대충 만들어서 주자"가 아니라 실제 운영 환경을 생각하고 있다.
- 5가지 기능을 전부 MVP에 넣겠다고 한 것. 범위를 줄이는 게 보통이지만, 엑셀을 완전히 대체하려면 실제로 이 정도는 필요하다. 현실적인 판단.
